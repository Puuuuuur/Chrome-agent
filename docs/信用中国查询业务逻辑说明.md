# 信用中国查询业务逻辑说明

这个文档只讲当前对外交付的稳定业务能力：

- `creditchina_query`

这里不再把 `private-api` 作为独立对外 skill 介绍。它现在只是 `creditchina_query` 内部优先使用的一条实现策略。

## 1. 用户怎么发起查询

### 页面聊天

最自然的方式是在页面里直接输入：

```text
请访问信用中国，用统一社会信用代码 91420000177570439L 完成一次查询，并告诉我最终结果。
```

### 固定接口

也可以直接调固定接口：

```json
{
  "credit_code": "91420000177570439L"
}
```

接口：

- `/api/creditchina/query`

### 显式 skills 接口

如果上层系统已经明确知道要跑哪个 skill，也可以直接调：

- `GET /api/playwright-agent/skills`
- `POST /api/playwright-agent/skills/run`

例如：

```json
{
  "skill_name": "creditchina_query",
  "input": {
    "credit_code": "91420000177570439L"
  }
}
```

## 2. 系统怎么做意图识别

聊天请求进入后，会先经过：

- `skills/registry.py`

优先级从高到低是：

1. 请求体里显式传了 `skill_name`
2. 系统判断消息是否是在请求标准信用中国查询
3. 如果都不是，就走 `browser_react`

### 什么情况下命中 `creditchina_query`

如果消息里提到了这些语义，通常会命中：

- 信用中国
- 查询 / 搜索
- 统一社会信用代码 / 信用代码
- 保存结果 / 固定查询 / 处理验证码

### 什么情况下不会命中

如果用户只是：

- 普通聊天
- 看页面结构
- 查 DOM
- 点按钮
- 排查挑战页

那通常会走：

- `browser_react`

## 3. 从输入到执行的完整链路

### 第 1 层：入口

入口文件：

- `app.py`

入口类型：

- 页面：`/playwright-agent/`
- 聊天：`/api/playwright-agent/chat`
- 固定接口：`/api/creditchina/query`
- 显式 skills：`/api/playwright-agent/skills/run`

### 第 2 层：同步调用封装

入口文件：

- `智能体调度.py`

职责：

- 把同步调用转成内部异步执行
- 把请求交给 `skills/registry.py`

### 第 3 层：意图识别与生命周期

核心文件：

- `skills/registry.py`

这里统一完成：

- skill 选择
- 输入 schema 校验
- 浏览器会话创建
- runtime 创建
- 结果收口
- session 持久化 / 失效标记

### 第 4 层：业务 skill

业务文件：

- `skills/skill_creditchina_query.py`

职责：

- 组织一次完整的信用中国查询
- 调用底层 runtime
- 把结果整理成统一输出

### 第 5 层：底层执行

技术底座：

- `tools/tool_browser_runtime.py`

这里负责：

- 浏览器会话接管 / 启动
- creditchina 页面准备
- private-api 请求
- DOM fallback
- 结果落盘

## 4. `creditchina_query` 是怎么跑的

这个 skill 的目标是：

- 尽量查到结果
- 把复杂链路收敛成一个稳定的对外能力

执行顺序是：

1. 准备浏览器会话
2. 调 `_prepare_creditchina_query_page_async()`
   - 进入查询上下文
   - 处理 cookie 预热、首页、挑战页续跑、验证码就绪等准备动作
3. 优先执行 `run_creditchina_private_api_flow_and_save_async()`
4. 如果 private-api 流成功：
   - 直接落盘
   - 返回结果
5. 如果 private-api 流失败：
   - 回退 DOM 查询流
   - 必要时处理页面验证码
   - 从 DOM 提取结构化结果
   - 再统一落盘

对用户来说，重点不是“到底走了哪条细链路”，而是：

- 你只调用一个稳定能力：`creditchina_query`
- 系统会自动优先选更稳的内部策略

## 5. 更底层到底做了什么

底层核心都在：

- `tools/tool_browser_runtime.py`

和信用中国直接相关的关键函数大致是：

- `_prepare_creditchina_query_page_async()`
  - 把页面推进到可查询状态
- `_capture_creditchina_runtime_state_async()`
  - 抽取当前页面和请求里的运行时参数
- `_run_creditchina_private_api_verify_async()`
  - 完成 API 验证码循环
- `_run_creditchina_private_api_query_async()`
  - 执行 private-api 主链路
- `_extract_creditchina_dom_result_async()`
  - 从 DOM 页面提取结构化结果
- `run_creditchina_private_api_flow_and_save_async()`
  - `creditchina_query` 内部优先使用的 private-api 流主入口
- `run_creditchina_query_and_save_async()`
  - 标准查询主入口

换句话说：

- `skills/registry.py` 决定“该跑哪个业务 skill”
- `tools/tool_browser_runtime.py` 决定“这个 skill 到底怎么把查询跑出来”

## 6. 已确认的底层接口事实

### 验证码

- `GET /private-api/verify/getVerify`
- `GET /private-api/verify/checkVerify`
- `code == 0` 视为校验成功

### 搜索

- `GET /private-api/catalogSearchHome`
- `status == 1` 视为成功

### 详情

- `GET /private-api/getTyshxydmDetailsContent`
- `status == 1` 视为成功

### 关键上下文

- `rcwCQitg` 是动态短效参数
- 搜索和详情接口依赖当前浏览器会话上下文
- 不建议脱离浏览器上下文独立重放核心请求

## 7. 结果怎么返回

不管内部走了 private-api 还是 DOM fallback，最终都会统一返回三层结果：

### 面向用户的摘要

- `reply`

### 面向业务的结构化结果

- `normalized`

通常包括：

- 企业名称
- 统一社会信用代码
- 状态
- 法定代表人
- 企业类型
- 成立日期
- 住所
- 登记机关

### 面向调试的完整结果

- `query_result`
- `agent_debug`
- `skill`

## 8. 结果文件落盘位置

默认目录：

- `法务测试/浏览器agent生成文件/信用中国查询/YYYY-MM-DD/单次运行目录/`

一次运行里通常会有：

- `*.json`
- `*.txt`
- `*.html`
- `*.png`

含义：

- `txt`
  - 给人快速读结果
- `json`
  - 给程序和调试看完整结构
- `html`
  - 页面源码快照
- `png`
  - 页面截图

## 9. 安全与脱敏

当前默认会做基础脱敏：

- 不保存完整 Cookie Header
- 不保存完整 `rcwCQitg`
- 不保存原始验证码值

已经处理过的字段包括：

- `captcha_guess`
- `verify_input`
- URL 里的 `rcwCQitg`

但要注意：

- `html` 仍然是原始页面快照
- 如果准备把结果发给别人，最好先自己检查一遍

## 10. 你应该怎么理解这套能力

最简单的理解方式是：

- 对外只有一个稳定查询能力：`creditchina_query`
- private-api 不是对外交付的第二个 skill，而是它内部优先使用的一条执行路径
- 你只需要调用一个稳定能力，不需要自己决定走 API 还是 DOM
